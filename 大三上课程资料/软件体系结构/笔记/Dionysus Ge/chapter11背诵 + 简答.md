# Chapter 11（Usability）

“架构对可用性的支持，一方面是允许用户在必要时采取主动（例如取消长时间命令、撤销已完成命令、对数据与命令做聚合操作）；另一方面，为了预测用户或系统响应，系统必须维护用户模型、系统模型与任务模型。”

------

## Chapter 11 必背定义（≥15个，含易错点）

> 格式：**术语(英文) — 一句话定义 — 关键点×2 — 常见误区×1**

1. **Usability（可用性/易用性）** — 关注用户完成任务的容易程度，以及系统提供的用户支持。— ①核心是“完成任务+支持”；②改进常体现为提升“用户对质量的感知”。— 误区：把 usability 等同于 UI 好看。
2. **User support（用户支持）** — 系统为帮助用户完成任务而提供的机制与能力。— ①可通过架构机制落地（撤销/取消/模型等）；②是 usability 的关键构成。— 误区：只当作“帮助文档/客服”。
3. **User’s perception of quality（用户质量感知）** — 用户对系统“好不好用/可靠/满意”的主观评价。— ①提升 usability 往往是最便宜、最容易的质量提升手段之一；②强调“感知”≠纯客观指标。— 误区：只盯客观性能指标，忽略用户体验。
4. **Learning system features（学习系统特性）** — 用户学习系统功能的难易程度。— ①对应“上手成本”；②常用“多久能熟练完成任务”来量化。— 误区：把“培训一下就会”当成系统天然易学。
5. **Using a system efficiently（高效使用）** — 用户熟练后完成任务的效率。— ①关注操作步骤/交互代价；②常与任务模型配合改进。— 误区：只优化新手学习，不管熟练用户效率。
6. **Minimizing the impact of errors（降低错误影响）** — 用户犯错时系统将损失/代价降到最低。— ①典型依赖 Undo/Cancel；②强调“可恢复”。— 误区：只做输入校验，不提供恢复能力。
7. **Adapting the system to user needs（适应用户需求）** — 系统可根据用户差异/情境调整支持方式。— ①常依赖 user model；②可表现为可配置、个性化、推荐等。— 误区：把“多加配置项”当成适应性全部。
8. **Increasing confidence and satisfaction（提升信心与满意度）** — 让用户对系统行为更可预期、更安心。— ①需要一致反馈；②与 system model 支持的“解释/反馈”相关。— 误区：只追求功能堆叠，忽略可预期性。
9. **Usability scenario（可用性场景）** — 用场景方式刻画“用户如何使用系统并达到可用性目标”。— ①可有具体场景示例（如两分钟上手）；②可结合 initiative 来选 tactics。— 误区：写成“系统要易用”这种不可测句子。
10. **Concrete usability scenario（具体可用性场景）** — 针对某系统的具体、可测描述。— ①例如“下载后两分钟可产出”；②必须能落到响应度量。— 误区：没有时间/成功标准。
11. **User initiative（用户主导）** — 人机对中由“用户”发起关键动作、决定交互推进。— ①与取消/暂停/撤销等强相关；②强调把控制权交给用户。— 误区：把“用户点按钮”都算 user initiative。
12. **System initiative（系统主导）** — 由系统主动判断/提示/辅助，推进交互。— ①依赖 task/user/system model；②目标是“预测并给出恰当反馈/帮助”。— 误区：把“自动弹窗提示”当成全部。
13. **Mixed initiative（混合主导）** — 用户与系统在不同步骤轮流主导。— ①可用性场景可混合两类 initiative；②架构上通常“既要可控也要可预测”。— 误区：混合=没有设计原则、想到啥做啥。
14. **Usability tactics（可用性战术）** — 架构层为达成可用性场景而采用的设计原语，按 initiative 分类讨论。— ①分“支持用户主导/支持系统主导”；②用于把“易用”落到结构机制。— 误区：把 tactics 当 UI 设计细节。
15. **Support User Initiative（支持用户主导）** — 一组让用户在关键时刻能掌控系统行为的战术集合。— ①典型：Cancel、Pause/Resume、Undo、Aggregate；②降低错误代价与操作负担。— 误区：只给按钮，不给底层语义与资源处理。
16. **Cancel（取消）** — 系统监听取消请求，终止被取消命令，释放资源并通知协作组件。— ①必须“终止+释放+通知”；②对长耗时操作尤其关键。— 误区：UI 上显示取消，但后台继续跑/资源不释放。
17. **Pause/Resume（暂停/继续）** — 临时释放资源以便重新分配，随后可恢复任务。— ①强调资源可回收；②保证恢复点一致。— 误区：只做“界面假暂停”，资源仍占用。
18. **Undo（撤销）** — 维护足够的系统状态信息，使系统可恢复到较早状态。— ①核心是“状态可回滚”；②减少错误影响。— 误区：只撤销显示层，不撤销真实状态/副作用。
19. **Aggregate（聚合操作）** — 将低层对象聚合成组，使一次操作作用于组，减少重复劳动。— ①降低操作“体力活”；②常对应批量处理/组操作。— 误区：只提供多选，不提供真正的组语义与命令应用。
20. **Support System Initiative（支持系统主导）** — 让系统“理解用户在做什么、用户是谁、系统将如何表现”，从而主动辅助。— ①依赖 task/user/system 三类模型；②用于预测响应并给反馈。— 误区：不建模，只靠 if-else 规则堆叠提示。
21. **Maintain Task Model（任务模型）** — 决定上下文，让系统理解用户意图并提供帮助。— ①用于判断“用户正在尝试什么”；②支撑更精准的引导与提示。— 误区：把“当前页面/状态”当成完整任务模型。
22. **Maintain User Model（用户模型）** — 显式表示用户对系统的知识、行为特征（如期望响应时间）等。— ①支持个性化与适应性；②用于预测用户对延迟/反馈的接受度。— 误区：用户模型=只存用户资料（姓名/年龄）。
23. **Maintain System Model（系统模型）** — 系统维护自身显式模型，用于判断期望行为并给恰当反馈。— ①用于解释“系统为什么这样”；②支持一致可预期的反馈。— 误区：把“日志/监控”当作系统模型本身。
24. **Quality attribute scenario 6-part template（质量属性场景六要素）** — Source/Stimulus/Environment/Artifact/Response/Response measure。— ①让需求可测；②可区分 general vs concrete。— 误区：缺 response measure，导致不可验收。
25. **Architectural tactic（架构战术）** — 影响某个质量属性响应的设计决策（原语）。— ①聚焦单一 QA 响应；②可被 pattern 打包。— 误区：把“战术”写成“架构风格/模式”级别。
26. **Seven categories of architectural design decisions（七类架构设计决策）** — Allocation of responsibilities、Coordination model、Data model、Resource management、Mapping、Binding time、Choice of technology。— ①“设计清单”常围绕这些类检查；②每类都有绑定时机选择。— 误区：只谈某一类（如技术选型），忽略其他类。

## 10个对比题答案（按考场写法）

### 1) structure vs view（至少3问）

**Q1：Structure vs View 区别？**
A：**Structure** 是系统由哪些元素及其关系构成的“客观组织”；**View** 是为满足某类干系人关注点而对一个或多个结构的**表示**（文档化表达）。

**Q2：一个 view 对应几个 structure？**
A：通常 **view 基于某个结构类型的实例**来表达；系统存在多种结构（模块、C&C、分配等），因此会有多种 view 来覆盖关注点。

**Q3：为什么要用 views 文档化？**
A：因为架构应以 views 形式被记录，以覆盖关键干系人关注点，并在项目时间线内可理解与可用（可沟通、可分析、可施工）。

### 2) module vs component（至少3问）

**Q4：Module vs Component 区别？**
A：**Module** 属于静态分解（代码/职责的组织单元）；**Component** 属于运行时结构（执行单元）的一部分。两者不是一一对应。

**Q5：模块与组件映射关系？**
A：模块在分解结构中可能表现为一个/部分/多个组件；一般映射是 **many-to-many**。

**Q6：培训新成员更适合看哪类 view？**
A：模块视图适合解释“谁做什么/团队分工”；C&C 视图适合解释“系统如何工作并完成任务”。

### 3) layer vs tier（至少2问）

**Q7：Layer vs Tier 区别（卷面版）？**
A：**Layer（层）** 是一种常见的模块类型模式：当 uses 关系严格单向时形成；层是一组相干功能。
**Tier（分层部署/执行分组）** 出现在分配/部署语境：在分布式部署中将系统划分为相对独立的执行结构分组（tier），用于组织组件的分组与交互。

**Q8：Tier 的关键约束是什么？**
A：在 multi-tier 模式下，一个软件组件**只属于一个 tier**（约束）。

### 4) pattern vs tactic（至少2问）

**Q9：Pattern vs Tactic 区别？**
A：**Pattern（架构模式）** 是反复出现的、能解决特定问题的一揽子结构化决策（元素类型+交互形式）；**Tactic（战术）** 是更原子的设计决策，用于影响某个质量属性响应。

**Q10：两者关系怎么写最加分？**
A：**Patterns are built from tactics**：如果 pattern 是“分子”，tactic 是“原子”；pattern 往往对其他质量属性中性或有弱点，需要用 tactics 补强。

## 6道简答题标准答案（完全应试写法）

### 1. 什么是 Usability？它包含哪些方面？

**答：**Usability 关注用户完成期望任务的容易程度以及系统提供的用户支持；其方面包括：学习系统特性、有效率地使用系统、最小化错误影响、适应用户需求、提升信心与满意度。

### 2. 为什么说提升 usability 常是“最便宜、最容易”的质量提升方式之一？

**答：**因为对 usability 的关注往往能以较低成本显著提升系统质量，尤其体现在提升用户对系统质量的感知；通过提供恰当的用户支持机制（如可取消、可撤销、可聚合操作等）即可直接改善使用体验。

### 3. 可用性战术的讨论为什么要引入 user initiative / system initiative / mixed initiative？

**答：**人机交互中可用性场景可由用户主导、系统主导或混合主导；用该区分可系统化选择战术：用户主导强调“用户可控”（cancel/undo/aggregate等），系统主导强调“系统可预测并能主动辅助”（维护任务/用户/系统模型）。

### 4. 解释 Cancel / Pause-Resume / Undo / Aggregate 四个“支持用户主导”的战术要点。

**答：**

- Cancel：系统监听取消请求；终止被取消命令；释放资源；通知协作组件。
- Pause/Resume：临时释放资源以便分配给其他任务，并支持恢复。
- Undo：维护足够的系统状态信息以恢复到先前状态。
- Aggregate：将低层对象聚合成组，使一次操作可作用于组，减少重复劳动。

### 5. 为什么支持“系统主导”需要维护 task/user/system 三类模型？分别做什么？

**答：**为预测用户或系统响应并提供恰当反馈，系统需要保持显式模型：任务模型用于确定上下文并推断用户意图；用户模型用于表示用户知识与行为（如期望响应时间等）；系统模型用于表示系统自身期望行为，从而给出正确反馈与解释。

### 6. 用“六要素场景模板”写一个可用性场景，并指出可用性战术如何支撑它。

**答：**

- Source：用户
- Stimulus：在执行一个耗时操作时发起取消/撤销请求
- Environment：正常运行（或高负载也可作为环境条件）
- Artifact：应用的某个命令/事务处理子系统
- Response：系统终止该命令并回收资源；必要时恢复到先前状态；给出反馈
- Response measure：取消响应时间、资源释放是否完成、状态恢复正确率/用户可继续操作的时间
  支撑战术：Cancel（终止+释放+通知）、Undo（恢复先前状态）；如涉及批量操作可用 Aggregate 降低操作负担。
